# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использовался датасет `S12-hw-dataset.csv`, прогнозировалась колонка `target`.
- Данные были разбиты на train / validation / test по времени: сначала 70% – train, далее 15% – validation, последние 15% – test.
- Сравнивались подходы: B1 (naive-last), B2 (moving-average), B3 (Ridge с лаговыми, rolling- и календарными признаками), R1 (GRU на оконном представлении ряда).
- Лучшей моделью по метрике `val_MAE` оказалась R1 (GRU), показавшая наименьшую ошибку и лучшую способность уловить сложные зависимости ряда.

## 2. Среда и воспроизводимость

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: ежедневная
- Горизонт прогноза: 1 шаг вперёд
- Размер датасета: 4306 наблюдений
- Диапазон дат: от 2025-01-01 до 2025-06-29
- Есть ли дополнительные внешние признаки: нет
- Комментарий: ряд демонстрирует явный недельный сезонный паттерн и некоторый тренд; присутствуют выбросы и пропуски отсутствуют после предобработки; прогнозирование осложнено сезонностью и возможной нестационарностью ряда.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: последовательное разбиение по времени без перемешивания.
- Какие доли или интервалы использовались: train 70%, validation 15%, test 15%.
- Почему `random split` здесь некорректен: перемешивание нарушило бы временную структуру, вызвав утечку информации о будущем, что недопустимо для прогнозирования временных рядов.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: день недели (day_of_week)
- Как обрабатывались пропуски после построения лагов/rolling: строки с NaN удалялись
- Использовалось ли масштабирование и где именно оно обучалось: для Ridge масштабирование признаков StandardScaler обучалось только на train; для GRU масштабирование таргета StandardScaler обучалось на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): прогноз равен последнему известному значению ряда.
- B2 (`moving-average`): прогноз по скользящему среднему за 7 дней. Валид.
- B3 (`ridge-lag-features`): Ridge на лаговых, rolling- и календарных признаках.
- R1 (`gru-forecast`): GRU с hidden_size=32, batch_size=32, window_size=7.

Дополнительно:

- Основная метрика выбора лучшей модели: val_MAE
- Дополнительные метрики: val_RMSE, val_MAPE
- `window_size` для `GRU`: 7
- Batch size: 32
- Максимальное число эпох: 20
- Optimizer: Adam
- Learning rate: 0.001
- Как сохранялась лучшая модель: по минимальному val_MAE через `state_dict()`.

## 7. Анализ

Наивный baseline B1 оказался достаточно сильным, отражая устойчивость ряда на предыдущий день. Moving-average B2 сгладил сезонные колебания, но потерял точность при резких скачках. Ridge B3 улучшила прогноз за счёт лаговых и rolling-признаков, особенно фиксируя недельную сезонность. GRU R1 показала наименьшую ошибку, эффективно учитывая короткие последовательности и тренды. Тренд и сезонность выражены умеренно, что ограничивает точность простых моделей. Лаговые и rolling-признаки помогли избежать утечек данных, которые могли бы возникнуть при random split. Выбор лучшей модели по validation принципиален, чтобы не использовать test для подгонки. Основные ошибки моделей связаны с резкими выбросами и локальными аномалиями ряда.

## 8. Итоговый вывод

В качестве базового подхода для данной задачи достаточно использовать B1 (naive-last) или B3 (Ridge) с лаговыми признаками. Простое baseline-прогнозирование подходит, когда ряд относительно стабилен и сезонность выражена явно. Переход к GRU имеет смысл при наличии сложных паттернов и трендов, которые сложно уловить табличными признаками. Главный вывод: корректная валидация временных рядов требует temporal split и строгого контроля утечек информации из будущего.
