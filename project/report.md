## 2. Постановка задачи и контекст

Опишите:

1. **Предметную область и задачу:**
   - Основной задачей является классификация заявок на кредит с использованием методов машинного обучения;
   - Потенциальными пользователями сервиса могут быть: финансовые организации и аналитики данных.

2. **Формулировку задачи в терминах ML/ИИ:**
   - На вход модели подаются характеристики заемщика и кредита, например: возраст клиента, пол, уровень образования, доход, стаж работы, тип жилья, сумма кредита, цель кредита, процентная ставка, кредитный рейтинг, история предыдущих просрочек и другие параметры.;
   - Целевой переменной является факт одобрения кредита;
   - К системе предъявляются следующие требования: высокая точность классификации, возможность обработки категориальных признаков, устойчивость к несбалансированным данным.

3. **Целевые метрики качества:**
   - Для оценки качества модели использовались следующие метрики: ROC-AUC, F1-score, Precision, Recall;
   - Метрики ROC-AUC, F1-score, Precision и Recall были выбраны, поскольку они позволяют комплексно оценить качество модели бинарной классификации в задаче кредитного скоринга.

## 3. Данные

Опишите, какие данные использует проект:

1. **Источник данных:**
   - Открытый датасет: https://www.kaggle.com/datasets/taweilo/loan-approval-classification-data;

2. **Структура данных:**
   - Основной набор данных представлен CSV-файлом: `data/loan_data.csv`, Размер датасета: 45 000 записей, 14 признаков;
   - Ключевые признаки: `person_age,person_gender,person_education,person_income,person_emp_exp,person_home_ownership,loan_amnt,loan_intent,loan_int_rate,loan_percent_income,cb_person_cred_hist_length,credit_score,previous_loan_defaults_on_file`. Целевая переменная: `loan_status`, где 1 — кредит одобрен, 0 — кредит не одобрен;
   - Числовые признаки: `person_age, person_income, person_emp_exp, loan_amnt, loan_int_rate, loan_percent_income, cb_person_cred_hist_length, credit_score`. Категориальные признаки: `person_gender, person_education, person_home_ownership, loan_intent, previous_loan_defaults_on_file`.

3. **Предобработка и EDA:**
   - В процессе подготовки данных были выполнены основные этапы разведочного анализа данных (EDA) и предобработки: разделение признаков на числовые и категориальные, анализ распределения призакнов, проверка данных на пропуски, анализ статистических характеристик признаков, выявление выбросов;
   - В ходе анализа было установлено: пропущенные значения в датасете отсутствуют; классы являются несбалансированными: класс 0 — 77.8%, класс 1 — 22.2%; в данных присутствуют выбросы, особенно в признаках: `loan_amnt, person_income, person_age`.

Основные шаги EDA лежат в `notebooks/01_dataset_eda.ipynb`.

## 4. Модели и подходы

1. **Базовые (baseline) модели:**
   - В качестве baseline-моделей были использованы Logistic Regression и Random Forest;
   - Результаты модели Logistic Regression на тестовой выборке: `ROC-AUC — 0.956; F1-score — 0.768; Precision — 0.790; Recall — 0.747; Accuracy — 0.900`;
   - Результаты Random Forest на тестовой выборке: `ROC-AUC — 0.974; F1-score — 0.830; Precision — 0.794; Recall — 0.870; Accuracy — 0.921`.

2. **Улучшенные модели и эксперименты:**
   - Основной моделью проекта стала CatBoostClassifier. Данный алгоритм был выбран благодаря хорошей работе с табличными данными и встроенной поддержке категориальных признаков без необходимости One-Hot Encoding;
   - Для подбора гиперпараметров применялся Optuna. В процессе оптимизации подбирались: `iterations, depth, learning_rate, l2_leaf_reg`;
   - Наиболее оптимальными гиперпараметрами оказались: `depth = 8; iterations = 997; learning_rate = 0.107; l2_leaf_reg = 8.34`;
   - Лучшая модель CatBoost показала следующие результаты на тестовой выборке: `ROC-AUC = 0.977; F1-score = 0.848; Precision = 0.891; Recall = 0.810; Accuracy = 0.936`;
   - Наиболее значимыми признаками оказались: `previous_loan_defaults_on_file, loan_percent_income, person_income, loan_int_rate`.

Эксперименты проводились в ноутубках: `notebooks/02_catboost_experiment.ipynb`, `notebooks/03_baseline_models.ipynb`.

## 5. Экспериментальный протокол и результаты

Здесь важно показать, что выбор финальной модели был осмысленным.

1. **Экспериментальный протокол:**
   - В рамках проекта использовалось разбиение данных на три непересекающиеся выборки: train — 27 000 объектов; validation — 9 000 объектов; test — 9 000 объектов;
   - Для всех моделей метрики рассчитывались: на validation выборке — для сравнения моделей и подбора параметров; на test выборке — для финальной оценки качества.

2. **Сравнение моделей по метрикам:**

   | Модель / конфигурация   | Описание                           | ROC-AUC | F1-score | Precision | Recall | Комментарий                     |
   | ----------------------- | ---------------------------------- | ------- | -------- | --------- | ------ | ------------------------------- |
   | Logistic Regression     | baseline, линейная модель          | 0.956   | 0.768    | 0.790     | 0.747  | Простая интерпретируемая модель |
   | Random Forest           | baseline, ансамбль деревьев        | 0.974   | 0.830    | 0.794     | 0.870  | Сильное улучшение recall        |
   | CatBoost (Optuna tuned) | бустинг с подбором гиперпараметров | 0.977   | 0.848    | 0.891     | 0.810  | Лучшая итоговая модель          |

3. **Выбор финальной модели:**
   - В качестве финальной модели была выбрана CatBoostClassifier с подобранными гиперпараметрами, оптимизированными через Optuna;
   - Причины выбора: наивысший ROC-AUC (0.977), наилучший F1-score (0.848), высокая precision (0.891), устойчивость к категориальным признакам без сложной предобработки.

## 6. Архитектура решения и сервис

Опишите, как из модели получился работающий сервис:

1. **Архитектура пайплайна:**
   - Система построена как асинхронный backend-сервис, разделяющий процессы аналитики данных и предсказаний модели;
   - Загрузка данных. Пользователь загружает CSV/XLSX файл через `/api/v1/analytic/` или `/api/v1/predict/`, файл сохраняется в S3, в БД создаётся запись о задаче
   - Предобработка. Сервис читает файл из S3, выполняется заполнение пропусков в датасете.
   - Предсказание. Загружается заранее обученная модель CatBoost, выполняется предсказание, после можно скачать результаты.

2. **API и endpoints:**
   - Реализованы эндпоинты для работы с пользователями, предсказаниями и анализом датасета. Подробнее написано в `README.md` в разделе 4.4;
   - Эндпоинт `/api/v1/predict/` предсказания принимает датасет и возвращает запись о результатах (подробнее можно увидеть в Swagger UI, `http://localhost:8000/docs`), после результат можно скачать через ручку `/api/v1/predict/{id}/report`.

3. **Технологический стек:**
   - Стек: `FastAPI, Pydantic, SQLAlchemy (async), asyncpg, CatBoost, Pandas, aioboto3, python-jose`;
   - Сервис рекомендуется поднимать в Docker с помощью команды `docker compose -p ai-project -f configs/docker-compose.yaml up -d`.

## 8. Ограничения и дальнейшая работа

- У проекта есть серьёзные ограничения в анализе датасета и предсказании, так как это CPU-bound задачи, и на данный момент они выполняются в асинхронном сервисе, что сильно замедляет время работы;
- что бы вы сделали, если бы было больше времени:
  - Добавление возможности обучения модели на своих данных;
  - Разбиение распределённого монолита на микросервисы: пользователи, обучение модели, предсказание датасета, анализ датасета;
  - Использование паттерна producer-consumer для сервисов с CPU-bound задачами, разделить их на оркестратора и воркер пул, взаимодействующими через брокер сообщений;
